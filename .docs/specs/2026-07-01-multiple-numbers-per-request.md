# Multiple Numbers Per Request â€” Design Document

## Problem / Background

`GET /random` returns exactly one random integer per call. Callers who need several numbers at
once (e.g., seeding a batch of test fixtures, running multiple dice rolls, populating a sample
dataset) must issue N separate HTTP requests, which is slower and puts unnecessary load on the
server. The API should let a caller request a batch of numbers in a single round trip.

## Goals

- Allow callers to optionally specify a `count` query parameter to receive multiple random
  numbers in one response.
- Maintain full backward compatibility â€” callers without `count` get the existing single-number
  response unchanged.
- Keep the existing no-consecutive-repeat and persistence guarantees intact for every number in
  a batch, not just single-number requests.

## Non-Goals

- Combining `count` with the (separately specified, not-yet-built) `min`/`max` range parameters â€”
  out of scope for this change; batches use the existing default [1, 100] range.
- Streaming or paginated delivery of large batches â€” a batch is returned as a single JSON array.
- Per-number distinctness within a batch beyond the existing no-consecutive-repeat rule (i.e.,
  non-adjacent duplicates within a batch are allowed, same as today across separate calls).

## Users / Personas

**API consumer** â€” any HTTP client calling `GET /random` who wants several random integers at
once instead of making repeated single-number calls (e.g., a test harness seeding fixtures, a
game requesting multiple dice rolls).

## Functional Requirements

**FR-1:** `GET /random` with no `count` parameter behaves exactly as today: returns one integer
in [1, 100] as `{ "data": { "number": N } }` with HTTP 200.

**FR-2:** `GET /random?count=N` where N is a valid integer between 1 and 100 (inclusive) returns
N random integers in [1, 100] as `{ "data": { "numbers": [n1, n2, ..., nN] } }` with HTTP 200.
This array shape is used whenever `count` is present, including `count=1`.

**FR-3:** Numbers within a batch are generated sequentially, applying the existing
no-consecutive-repeat rule across the entire sequence: each generated number differs from the
immediately preceding one, whether that preceding number came from earlier in the same batch or
from the last number generated by any prior request (the server-global `lastNumber`).

**FR-4:** After a batch completes, the server-global `lastNumber` is updated to the last number
in the batch, so the no-consecutive-repeat guarantee carries forward into the next request
(single or batch).

**FR-5:** Every number generated as part of a batch is persisted to the history store
individually, in generation order, using the same persistence mechanism as single-number
requests (see `2026-06-30-persist-random-numbers.md`). `GET /random/history` reflects all
numbers from a batch as separate entries.

**FR-6:** If `count` is provided but is not a valid positive integer (e.g., a decimal, a string,
zero, a negative number, or an empty string), the endpoint returns HTTP 400 with:
`{ "error": { "type": "validation", "message": "count must be a positive integer" } }`

**FR-7:** If `count` exceeds 100, the endpoint returns HTTP 400 with:
`{ "error": { "type": "validation", "message": "count must not exceed 100" } }`

**FR-8:** A batch request that fails validation (FR-6, FR-7) generates and persists nothing â€”
no partial batches are written on a rejected request.

## Non-Functional Requirements

- No new production dependencies; batch generation reuses existing random-number and persistence
  logic in a loop.
- All new query-parameter branches (valid batch, invalid type, zero/negative, over-cap) covered
  by automated tests.
- A full 100-number batch completes well within normal request latency expectations (no
  artificial delay between generations).

## Acceptance Criteria / Success Metrics

| Request | Expected |
|---|---|
| `GET /random` | `{ "data": { "number": N } }`, HTTP 200 (unchanged) |
| `GET /random?count=5` | `{ "data": { "numbers": [n1..n5] } }`, HTTP 200, 5 integers in [1,100] |
| `GET /random?count=1` | `{ "data": { "numbers": [n1] } }`, HTTP 200 (array, not scalar) |
| `GET /random?count=100` | 100 integers, HTTP 200 |
| `GET /random?count=101` | HTTP 400, `validation` error, "must not exceed 100" |
| `GET /random?count=0` | HTTP 400, `validation` error |
| `GET /random?count=-3` | HTTP 400, `validation` error |
| `GET /random?count=abc` | HTTP 400, `validation` error |
| `GET /random?count=2.5` | HTTP 400, `validation` error |
| `GET /random?count=5` then `GET /random/history` | history includes all 5 new numbers, most recent first |
| Two adjacent numbers within one batch | never equal (no-consecutive-repeat holds within batch) |
| Last number of batch vs. first number of next request | never equal (no-consecutive-repeat holds across requests) |

## Scope

**In:**
- `count` query parameter parsing and validation on `GET /random`
- Array response envelope when `count` is present
- Sequential generation reusing the existing no-consecutive-repeat and persistence logic per
  number in the batch
- Upper bound of 100 numbers per request

**Out:**
- Combining `count` with range (`min`/`max`) parameters
- Any change to `GET /random/history`'s response shape (it already returns an array)
- Rate limiting or authentication
- Streaming responses

## Key Decisions & Rationale

**`count` as the parameter name, capped at 100:** `count` is unambiguous and self-descriptive.
A cap prevents a single request from generating an unbounded number of rows (protects response
size and the persistence store from a single abusive/mistaken request). 100 mirrors the existing
default range ceiling, keeping the API's numeric conventions consistent.

**Array response only when `count` is present, even for `count=1`:** Keeps the response shape
predictable based on whether the caller opted into batching, rather than being value-dependent
(a caller passing `count=1` explicitly wants array-shaped output, e.g. for uniform client-side
handling). Omitting `count` entirely is the only way to get the legacy scalar shape, so nothing
breaks for existing callers who never set `count`.

**No-consecutive-repeat applies across the whole sequence, chained from `lastNumber`:** Treats a
batch as N sequential generations rather than a special case, reusing
`2026-06-30-no-consecutive-repeat.md`'s existing invariant instead of introducing a second rule
that only applies to batches.

**All-or-nothing validation (FR-8):** Validating before generating avoids partially-persisted
batches when a request is rejected, keeping the history store's contents always traceable to a
successful request.
